# Narrative Authority — 사이트 전체 재설계 스펙

## 목적

dasomel.github.io를 평면적 블로그에서 "내러티브 + 권위"가 결합된 전문가 포트폴리오로 전환한다. 방문자가 깊이 탐색할 동기를 만들고, 스토리텔링(50%)과 전문성 어필(50%)을 균형 있게 전달한다.

## 타깃 오디언스

| 그룹 | 비중 | 핵심 니즈 |
|---|---|---|
| 채용 담당자 / 기업 | 40% | 기술 역량, 경력 깊이, 성과 증거 |
| 오픈소스 커뮤니티 / 동료 엔지니어 | 40% | 기술 콘텐츠, 협업 가능성, 프로젝트 맥락 |
| 학생 / 강의 수강생 | 20% | 교수님 배경, 학습 자료 접근 |

## 디자인 원칙

- **미니멀 + 권위**: 현재의 깔끔한 미니멀 톤을 유지하면서, 숫자·로고·배지로 전문성을 시각적으로 전달
- **탐색 동기 부여**: 각 섹션이 다음 섹션으로의 호기심을 유발하는 구조
- **문제→해결 스토리**: 프로젝트와 OSS 기여를 맥락 없는 나열이 아닌 "왜 만들었나"로 전달
- **근사치 활용**: 정확한 숫자 없이도 "20+", "11yr" 등 근사치로 임팩트 전달

## 1. 홈페이지

### 1.1 Hero 섹션

**현재**: 제네릭 헤드라인 "클라우드 네이티브 인프라를 코드로" + 기술 칩 나열

**변경**:
- 헤드라인 개인화: "11년간 클라우드 인프라를 만들고, 가르치고, 나눕니다"
- 상태 뱃지에 역할 2개 병기: "Cloud & DevOps Engineer · Adjunct Professor"
- 서브텍스트: "K-PaaS 핵심 컨트리뷰터 · OPA/OPDC 커뮤니티 리더 · 한국공학대학교 겸임교수"
- CTA 버튼: "프로젝트 보기" + "발표 이력" (현재 "블로그 읽기" → "발표 이력"으로 변경)
- 기술 칩은 유지

### 1.2 Impact Bar (신규)

Hero 바로 아래, 전체 너비 3컬럼 바:

| 항목 | 값 | 설명 |
|---|---|---|
| 컨퍼런스 발표 | 20+ | seminars 콘텐츠 수에서 동적 계산 |
| 클라우드 경력 | 동적 | `현재 연도 - 2013`으로 자동 계산 |
| OSS 프로젝트 | 5+ | projects 콘텐츠 수에서 동적 계산 |

CSS: 상하 `border` 구분선, 중앙 정렬, 에메랄드 숫자, 회색 레이블. 값은 하드코딩이 아닌 콘텐츠에서 동적으로 계산.

### 1.3 Featured Work 섹션

**현재**: 2x2 프로젝트 카드 그리드 (제목 + 설명 + 태그)

**변경**: 주요 프로젝트 2개를 스토리 카드로 (featured 프로젝트가 3개 이상이면 최대 3개까지):
- 각 카드에 "문제→해결" 한 줄 맥락 추가
- 에메랄드 "→" 링크로 상세 페이지 연결
- 나머지 프로젝트는 "전체 보기 →" 링크

### 1.4 Speaking Highlights 섹션 (신규)

**현재**: 없음 (OSS 기여 섹션이 있었음)

**변경**: 주요 발표 2개를 하이라이트 (featured 세미나가 3개 이상이면 최대 3개까지):
- 컨퍼런스 약어 텍스트 배지 (32x32, 예: "OID", "KCD" — 이미지 로고가 아닌 텍스트 기반) + 컨퍼런스명 + 발표 제목 + 연도
- "전체 발표 이력 →" 링크
- OSS 기여 섹션은 홈에서 제거 (About으로 이동)

### 1.5 Latest Posts 섹션

**현재**: 최근 4개 포스트 리스트

**변경**:
- 추천글 1개를 상단에 핀 처리 (에메랄드 보더)
- 나머지 최근 3개 포스트
- 각 포스트에 읽기 시간 표시

## 2. About 페이지

### 2.1 프로필 헤더

**현재**: 이름, 역할 1개, 위치, GitHub 링크

**변경**:
- 역할 표시를 3개로 확장: "Cloud Engineer · Professor · Community Leader"
- GitHub + LinkedIn 링크 (LinkedIn이 있다면)

### 2.2 내러티브 인트로 (신규)

왼쪽 에메랄드 보더의 인용 스타일 블록:

> "2013년 eGovFrame 커뮤니티에서 시작해, 클라우드 네이티브 인프라를 만들고 운영해왔습니다. K-PaaS의 핵심 컨트리뷰터로 참여하며, OPA와 OPDC 커뮤니티를 이끌고, 현재는 한국공학대학교에서 후배 엔지니어를 가르치고 있습니다."

사용자가 실제 내용을 수정할 수 있도록 콘텐츠는 하드코딩이 아닌 편집 가능한 형태로 구현한다.

### 2.3 역할 카드 (재구성)

**현재**: 경력 2개 항목 (교수, 엔지니어)

**변경**: 3개 역할 카드, 각각 왼쪽 에메랄드 보더:

1. **Cloud & DevOps Engineer** (2013~) — K-PaaS 핵심 컨트리뷰터, Harbor/Terraboard 포크, Kubernetes 인프라 설계/운영
2. **겸임교수** (2024~) — 한국공학대학교 IT경영전공, SW프레임워크 강의
3. **커뮤니티 리더** (2018~) — OPA (Open cloud Platform Alliance), OPDC (Open Platform Developer Community) 리더

### 2.4 전문가 활동 (신규, ~10건)

카테고리 필터 탭: 자문 / 심사 / 멘토링

각 항목: 기관명 + 활동명 + 연도 (연도 역순)

예시 형태:
```
◆ 정보통신산업진흥원(NIPA) 자문위원     2023
◆ K-PaaS 기술 자문                       2022
```

데이터 소스: `lib/data/about.ts`에 타입 정의와 데이터 배열을 별도 관리 (About 페이지 300줄 초과 방지). 사용자가 실제 데이터를 입력해야 함.

### 2.5 수상 이력 (신규, ~10건)

연도 역순 리스트. 주요 수상(장관상 등)은 하이라이트 처리 (볼드 + 트로피 아이콘 또는 에메랄드 텍스트).

예시 형태:
```
🏆 과학기술정보통신부 장관상              2024
🏆 NIA 원장상                            2023
```

데이터 소스: `lib/data/about.ts`에서 전문가 활동과 함께 관리. 사용자가 실제 데이터를 입력해야 함.

### 2.6 연구보고서 (신규, 3~4편)

NIA 의뢰 명시. 보고서 제목 + 의뢰 기관 + 연도 + PDF/요약 링크 (있는 경우).

예시 형태:
```
📄 K-PaaS 기술 동향 보고서               NIA · 2024
📄 클라우드 플랫폼 분석                   NIA · 2023
```

### 2.7 컨퍼런스 월 (신규)

발표했던 컨퍼런스/행사명을 배지로 나열. `seminars` 콘텐츠에서 컨퍼런스명을 자동 추출하거나, 별도 데이터로 관리.

배지 스타일: `px-2 py-1 bg-gray-50 border border-gray-200 rounded-md text-xs`

### 2.8 오픈소스 기여 (개선)

**현재**: 프로젝트명 + 한 줄 설명

**변경**: "문제→해결" 한 줄 스토리 형태:
- Harbor: "ARM 아키텍처 미지원 문제 → 직접 포크하여 해결"
- Terraboard: "구버전 고착 문제 → 최신 Terraform 버전 적용 포크"

### 2.9 섹션 순서

프로필 헤더 → 내러티브 인트로 → 역할 카드 → 전문가 활동 → 수상 이력 → 연구보고서 → 컨퍼런스 월 → 오픈소스 기여

## 4. Work 페이지

### 4.1 페이지 헤더

- 제목: "Work"
- 서브텍스트: "오픈소스 프로젝트와 인프라 도구"

### 4.2 태그 필터 (신규)

필터 칩: All / Kubernetes / K-PaaS / Terraform / Fork 등

`tags` 프론트매터에서 자동 추출. 클릭 시 URL 쿼리 파라미터 또는 클라이언트 사이드 필터링 (SSG이므로 클라이언트 사이드 권장).

### 4.3 케이스 스터디 카드 (주요 프로젝트)

주요 프로젝트 2~3개:
- 프로젝트명 + 타입 뱃지 (Fork/Project)
- "문제 | 해결" 2컬럼 레이아웃
- 태그 목록
- GitHub 링크

"주요" 선정: 프론트매터에 `featured: true` 플래그 또는 `type` 필드 활용.

### 4.4 컴팩트 카드 (나머지 프로젝트)

1줄 형태: 프로젝트명 + 한 줄 설명 + 외부 링크 화살표

### 4.5 Projects/Forks 구분 제거

현재의 `type` 기반 섹션 분리를 제거하고, 타입 뱃지로 대체. 모든 프로젝트가 하나의 목록에서 중요도 순으로 표시.

### 4.6 프론트매터 변경

기존 `type: project | fork` 유지하되, `featured: true` 필드 추가:

```yaml
---
title: Harbor ARM Fork
type: fork
featured: true
problem: "ARM 아키텍처 미지원"
solution: "멀티 아키텍처 빌드 파이프라인 구성"
---
```

## 8. 데이터 요구사항

사용자가 직접 입력해야 하는 데이터:

| 항목 | 위치 | 형태 |
|---|---|---|
| 전문가 활동 (~10건) | `lib/data/about.ts` | `{ category: 'advisory'|'review'|'mentoring', title, org, year }` |
| 수상 이력 (~10건) | `lib/data/about.ts` | `{ title, org, year, highlight?: boolean }` |
| 연구보고서 (3~4편) | `lib/data/about.ts` | `{ title, client, year, url?: string }` |
| 내러티브 인트로 텍스트 | `lib/data/about.ts` | `{ ko: string, en: string }` |
| 프로젝트 problem/solution | 각 프로젝트 `.md` 프론트매터 | `problem`, `solution` 필드 |
| 세미나 conference 필드 | 각 세미나 `.md` 프론트매터 | `conference` 필드 |
| featured 플래그 | 포스트/프로젝트/세미나 프론트매터 | `featured: true` |

## 9. 기술적 고려사항

- **SSG 호환**: `output: 'export'` 유지. 태그 필터는 클라이언트 사이드로 구현
- **i18n**: 모든 신규 텍스트는 `messages/ko.json`, `messages/en.json`에 추가. 데이터 배열은 `isEn` 분기 유지
- **기존 디자인 토큰 활용**: `var(--accent)`, `var(--border)`, `var(--surface)` 등 CSS 변수 그대로 사용
- **컴포넌트 분리**: 300줄 초과 방지. About 페이지는 섹션별 컴포넌트로 분리
- **반응형**: 모바일 우선. Impact Bar는 모바일에서 3컬럼 유지 (값이 짧으므로 가능)
